# 产品进化路线：从二手车 API 到 Agent 协作网络

## 来源启发

本路线吸收了两类 Agent 产品经验：

- 爱马仕智能体：自主记忆、自主建技能、自我修复、经验反哺
- Agent 社交网络：长期在线、A2A 协作、Memory OS、协议层、管理者监督

目标不是堆功能，而是把二手车业务做成一个能持续学习、可监督、可治理的工具型 Agent 网络。

## 我们的产品定位

当前项目不是二手车金融交易平台，而是：

- 买家 Agent 的需求大厅
- 车商 Agent 的车源响应系统
- 平台 Agent 的撮合与调度系统
- 管理者的人类监督台

核心体验应该是：

```text
人提出目标，Agent 负责探索、协商、记录，人最后监督和决策。
```

## 一、用户体验原则

### 1. 从“手动搜索车源”变成“挂需求等 Agent 跑”

用户不应该反复筛选、私聊、砍价。用户只需要描述：

- 预算
- 城市
- 用途
- 风险偏好
- 品牌偏好
- 不接受条件

系统负责：

- 生成结构化需求
- 持续监控车源池
- 找不到时通知车商 Agent 补车
- 找到后发起询价
- 生成风险摘要与建议方案

### 2. 从“黑箱推荐”变成“透明协商过程”

每次匹配、询价、议价都要留下可解释记录：

- 为什么匹配这辆车
- 哪些条件满足，哪些条件不满足
- 为什么建议加预算或换品牌
- 车商 Agent 如何回应
- 最终是否达成沟通意向

用户看到的不是一个结果，而是一条可追踪的 Agent 工作流。

### 3. 从“卖车广告”变成“车档 + 信誉 + 响应能力”

车商端不只是发布车源，还要被评价：

- 车况透明度
- 响应速度
- 报价合理性
- 历史履约表现
- 被举报和审核记录

高质量车商获得更多需求响应机会。

## 二、Agent 架构原则

### 1. 三类核心 Agent

买家 Agent：

- 代表买家利益
- 管理预算、偏好、风险底线
- 发需求、查匹配、询价、提出报价

车商 Agent：

- 代表车商利益
- 管理车源、利润空间、响应优先级
- 发布车源、回复询价、评估报价

平台调度 Agent：

- 不代表任何一方
- 负责撮合、监督、记录、风控
- 判断下一步应该让谁行动

### 2. A2A 的真正含义

A2A 不是两个聊天机器人互相闲聊，而是：

```text
买家 Agent -> 平台状态 -> 车商 Agent -> 平台状态 -> 买家 Agent
```

平台必须保存状态，而不是依赖上下文窗口。

### 3. 先 HTTP 工具化，再 MCP 标准化

当前阶段：

- HTTP API 已经足够让 Qclaw / WorkBuddy 测试
- 先验证业务闭环

下一阶段：

- 把高频动作封装成 MCP 工具
- 让 Agent 不需要理解全部 API，只调用稳定工具

推荐 MCP 工具：

- `create_buyer_profile`
- `publish_demand`
- `publish_car`
- `find_matches`
- `send_inquiry`
- `send_counter_offer`
- `create_deal_intent`
- `record_agent_observation`

## 三、记忆与成长机制

### 1. 每轮协商都要沉淀记忆

应记录：

- 用户真实偏好
- 用户反复拒绝的条件
- 成交/失败原因
- 车商响应质量
- 常见议价策略

不要只存聊天记录，要沉淀结构化经验。

### 2. 后台复盘机制

参考爱马仕思路，平台应定期复盘最近 N 次任务：

- 哪些需求反复匹配失败
- 哪些车商响应好
- 哪些报价策略有效
- 哪些流程需要变成新技能

复盘不应该打断主流程，应作为后台任务运行。

### 3. 自主建技能

当某类流程反复出现时，应沉淀成技能：

- 低预算 SUV 寻车技能
- 北京通勤家用车筛选技能
- 新能源保值率解释技能
- 车商报价合理性判断技能
- 事故/调表风险摘要技能

技能要能被 Agent 复用，也要能被管理者审核。

## 四、管理者体验

管理者不是看数据库，而是看 Agent 网络运行情况。

管理后台应提供：

- 今日新增需求
- 今日新增车源
- 匹配成功率
- 无匹配需求列表
- 最活跃买家 Agent
- 最活跃车商 Agent
- 被举报车商
- 异常报价
- 失败协商原因

关键目标：

```text
让管理者知道系统为什么这么做，以及哪里需要人工介入。
```

## 五、安全与合规护栏

### 1. 个人主体小程序边界

生产路径不能出现：

- 支付
- 托管
- 贷款
- 利率
- 金融推荐
- 收益承诺
- 平台担保成交

### 2. Agent 行为边界

Agent 可以：

- 推荐沟通方案
- 解释价格区间
- 提醒风险
- 生成询价/议价话术

Agent 不能：

- 代替用户付款
- 承诺车况绝对真实
- 承诺成交
- 伪造检测报告
- 泄露其他用户隐私

### 3. 私有核心保护

当前开源是为了安装和测试。正式商业化前必须拆分：

- 公开：Skill、OpenAPI、SDK、前端壳、非敏感协议
- 私有：匹配排序、车商权重、风控、反作弊、议价策略、管理规则、数据库

## 六、近期落地顺序

### 阶段 1：双 Agent 人工调度测试

当前正在做：

- Qclaw 扮演买家 Agent
- WorkBuddy 扮演车商 Agent
- 人工复制消息做调度
- CloudBase 保存状态
- `record-and-reward` 接口把录档、链式存证和积分奖励串成一个动作
- Agent 事件接口记录 Qclaw / WorkBuddy 的关键观察和执行结果
- 管理者漏斗接口显示需求、车源、协商和未匹配需求

目标：

- 跑通买家需求、车商上车、匹配、询价、议价、达成意向

### 阶段 2：平台自动调度器

把人工复制粘贴变成后端任务：

- 匹配为空 -> 通知车商 Agent 补车
- 新车上架 -> 重新匹配未满足需求
- 买家询价 -> 生成车商回复任务
- 车商报价 -> 生成买家决策建议

### 阶段 3：接入腾讯混元

混元先做低风险任务：

- 总结车况
- 生成询价话术
- 解释匹配原因
- 生成管理者日报
- 判断下一步调度动作

不要一开始就让模型直接决定高风险业务规则。

### 阶段 4：MCP 化

把 HTTP API 封装成 Agent 工具包，让不同 Agent 平台都能稳定调用。

### 阶段 5：私有核心拆分

当真实用户和真实车商进入后，迁移核心算法到腾讯云私有服务，只保留公开协议层。

## 七、产品判断标准

每次迭代都问四个问题：

1. 用户是不是更少手动操作？
2. Agent 是不是能更主动推进？
3. 管理者是不是更清楚系统在发生什么？
4. 核心规则是不是仍然安全、合规、可控？

如果答案不是，这个功能就不优先。
